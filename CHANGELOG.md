### Added
- Phase 0 scaffold: README with tagline + status badge, expanded `.gitignore` covering Python/macOS/VSCode/secrets, `CHANGELOG.md`, `docs/benchmarks.md` placeholder.
- Phase 1 (deploy-first): `scripts/ec2_userdata.sh` cloud-init bootstrap (installs Docker on AL2023, enables daemon, adds `ec2-user` to docker group). Verified end-to-end with a `c6i.large` instance + `hello-world` smoke test; instance stopped after verification to avoid spend.
- Phase 1 service skeleton: `moderation_engine.model.ToxicityClassifier` wrapping `unitary/toxic-bert`; `moderation_engine.api` FastAPI app with `POST /classify` and `GET /health`; pydantic-settings-driven `config.py`; structlog JSON logging via `_logging.py`. Smoke tests in `tests/test_smoke.py` (toxic-input flagged, clean-input not flagged, all six labels returned). All checks pass; local `curl` against `/classify` returns expected scores.
- Phase 1 containerization: multi-stage `Dockerfile` (uv builder + slim runtime, non-root user, `HEALTHCHECK` on `/health`, toxic-bert weights pre-baked into `/opt/hf-cache`, `TRANSFORMERS_OFFLINE=1` + `HF_HUB_OFFLINE=1`). `.dockerignore` strips data, tests, scripts, docs. `docker-compose.yml` for local dev with healthcheck and port mapping. `pyproject.toml` pins torch to the PyTorch CPU wheel index on Linux only via `[[tool.uv.index]]` + `marker = "sys_platform == 'linux'"`, removing 14 CUDA/nvidia-* packages and triton from the Linux lockfile. Final single-platform `linux/arm64` image is 822 MB (target: < 1.5 GB). Classification output is byte-identical to the host service.
- Phase 1 deploy on AWS: ECR repository `086002003795.dkr.ecr.us-east-1.amazonaws.com/moderation-engine` (scan-on-push). Multi-arch image pushed under tags `latest` and `phase1-baseline` (amd64 ~873 MB + arm64 ~823 MB). IAM role `moderation-engine-ec2` + instance profile of the same name attached to `i-0c8898bf51c280eb4` (`AmazonEC2ContainerRegistryReadOnly`), so the host can `aws ecr get-login-password` against IMDS. End-to-end verified: started instance, `docker pull`, `docker run -d -p 8000:8000`, `curl http://<public-dns>:8000/classify` from laptop returns scores byte-identical to local Mac (arm64 container) and host (uncontainerized). Instance stopped after verification.
- Phase 1 accuracy floor: `scripts/eval_baseline.py` runs `unitary/toxic-bert` over the 63,978-row Jigsaw scored test split at threshold 0.5 and reports per-label F1 / precision / recall / ROC-AUC. Local MPS run, batch size 32, ~33 min wall (31.8 samples/s). Macro-F1 **0.6101**; ROC-AUC ≥ 0.97 on every label (the model ranks well; the 0.5 threshold is recall-heavy and will be tuned per-label in Phase 2). Raw per-row predicted probabilities committed to `docs/baseline_predictions.parquet` (4.0 MB) for parity checks against ONNX / INT8 exports. Numbers transcribed into `docs/benchmarks.md` accuracy floor + Phase 2 optimization journey table.
- Phase 1 load-test harness: `locustfile.py` (closed-loop `ClassifyUser`, 1000-row stable Jigsaw sample seeded at startup, no think time), `scripts/run_locust_sweep.sh` (1/5/10/25/50/100-user sweep × 60 s, gated on `/health`), `scripts/summarize_locust.py` (parses per-level CSVs into a markdown table). Sweep not yet executed against EC2 — pending instance restart.
- Phase 1 latency baseline: ran the full 1/5/10/25/50/100-user sweep against `i-0c8898bf51c280eb4` (instance restarted, container auto-came-up via `--restart unless-stopped`, sweep run from laptop, instance stopped after). Numbers landed in `docs/benchmarks.md`: single-request p99 **860 ms**, throughput saturates at **~7 req/s** with no batching, p99 explodes linearly past saturation (27 s at 100 users). Zero failures across all levels. This is the diagnostic shape Phase 2 targets — ONNX/INT8 to lower the floor, dynamic batching to lift the throughput ceiling. **Phase 1 exit criteria met.**
- Phase 2 (Opt 1) ONNX backend — accuracy parity: refactored `moderation_engine.model` into a backend-agnostic protocol + `build_classifier` factory backed by concrete `PyTorchToxicityClassifier` and `ONNXToxicityClassifier` classes under `moderation_engine/backends/`. `scripts/export_onnx.py` exports `unitary/toxic-bert` to ONNX via `optimum` (438 MB fp32, gitignored). `scripts/parity_check.py` runs both backends on the same N rows and asserts max abs prob diff ≤ tolerance. `colab/eval_onnx.ipynb` ran the full 63,978-row Jigsaw eval on Colab T4 (CUDAExecutionProvider). **Macro-F1 0.6101 — byte-identical to PyTorch at four decimals**; full-set element-wise diff against `baseline_predictions.parquet` shows max 3.76e-06, mean 1.13e-08, 0 rows over the 1e-4 tolerance. ONNX is a lossless numerical equivalent — any latency win is free.
- Phase 2 (Opt 1) ONNX backend — latency sweep: locked Phase 1 sweep (1/5/10/25/50/100 users × 60 s) re-run against the ONNX-backed container on `i-0c8898bf51c280eb4`. Pushed multi-arch image to ECR as `phase2-onnx` + `latest` (amd64 521 MB compressed, down from 873 MB on `phase1-baseline`), pulled on EC2, swapped containers, ran sweep from laptop, stopped instance after. Headline: single-request **p50 −25%** (160 → 120 ms), **throughput +30%** (4.3 → 5.6 req/s at 1 user; +24-40% across the curve, peaking at ~8.8 req/s vs Phase 1's ~7.1). **Surprise**: p99 *widened* at mid concurrency (notably +90% at 25 users) — root cause is ONNX Runtime's CPU provider defaulting to `intra_op_num_threads = num_cores`, which is a win at a single in-flight request but causes thread contention when multiple requests are in flight on a 2-vCPU host. PyTorch was effectively single-threaded per inference (slower p50, tighter tail). The throughput ceiling stays single-process-bound — Opt 3 (dynamic batching) is what lifts it. Zero failures across all 6 levels. Numbers + delta table + "what surprised" written into `docs/benchmarks.md`; optimization-journey row "+ ONNX Runtime" now populated. Phase 1 locust CSVs archived under `docs/locust/phase1-baseline/` for diff. **Phase 2 Opt 1 complete.**
- Phase 2 (Opt 2) INT8 dynamic quantization — accuracy eval: `scripts/quantize_onnx.py` applies `onnxruntime.quantization.quantize_dynamic` (`QInt8`, weights only) to the fp32 ONNX export, producing `models/onnx-toxic-bert-int8/` (105 MB, **−76%** vs 439 MB fp32). `scripts/eval_baseline.py` gained `--onnx-dir` + `--out-prefix` flags plus sort-by-token-length so most batches use minimal padding (3.6× throughput on the 320-row check). Plan called for Colab T4 — didn't work: ORT has no CUDA kernel for `MatMulInteger` / `DynamicQuantizeLinear`, so QInt8 ops fall back to CPU per-op with constant CPU↔GPU tensor copies (first run estimated 13 h). Colab CPU was worse (77 s/batch) — free-tier Xeons predate AVX-VNNI, so INT8 matmul gets no SIMD acceleration. Ran locally on M3 Pro instead (ARM SDOT is the ARM-side AVX-VNNI equivalent): 76.5 samples/s, full 64k eval in 14 min. **Macro-F1 0.6101 → 0.6146 (+0.0045)**; F1 up on 5/6 labels, `severe_toxic` −0.0027 (367 positives, inside noise). Full-set decision diff against `baseline_predictions.parquet`: **1,939 / 383,868 binary decisions flip at thr 0.5 (0.505%)**, 1,858 / 63,978 rows have any flip (2.90%). Mechanism: precision up and recall down on every label — INT8 mildly compresses logits toward 0, FP32 baseline was precision-limited, so the trade-off lands on the right side of the bias/variance curve. Discard branch in plan.txt does not trigger. Deleted `colab/eval_int8.ipynb` (Colab dead end). Saved `onnx_int8_cpu_only` memory so this hardware-feature mismatch doesn't bite again.
- Phase 2 (Opt 2) INT8 dynamic quantization — latency sweep: same locked protocol against the INT8-backed container on `i-0c8898bf51c280eb4`. Multi-arch image pushed to ECR as `phase2-int8` + `latest` (**173 MB compressed, −67% vs `phase2-onnx`**), pulled on EC2, container swapped, sweep run from laptop, instance stopped after. Headline: **p50 −50 to −66% across the entire concurrency curve** (e.g., 1u 120→60 ms, 25u 1,900→640 ms), **p99 −43 to −63%** (1u 930→430 ms, 25u 19,000→7,000 ms), **throughput +120 to +191%** (1u 5.6→12.3 req/s; ceiling lifts from ~9 to ~23 req/s). The ONNX FP32 p99 regression at mid concurrency is gone — at 25 users, ONNX FP32 had p99 +90% vs PyTorch baseline; INT8 turns that into p99 −30% vs baseline at the same level. The threading model didn't change; each request just finishes ~3× faster so contention costs are paid in a much smaller window. Zero failures across all 6 levels. Plan expected "2-3x latency reduction" — actual win is bigger than that on every recorded column. Container size compounds the Opt 1 win: `phase1-baseline` 873 MB → `phase2-onnx` 521 MB → `phase2-int8` 173 MB (cumulative −80%). Opt 1 CSVs archived under `docs/locust/phase2-onnx/`, INT8 CSVs under `docs/locust/phase2-int8/`. Numbers + delta table + "what surprised" written into `docs/benchmarks.md`; optimization-journey row "+ INT8 dynamic quantization" now populated. **Phase 2 Opt 2 complete.**
