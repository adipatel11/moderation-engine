Multi-arch image pushed under tags `latest` and `phase1-baseline` (amd64 ~873 MB + arm64 ~823 MB). IAM role `moderation-engine-ec2` + instance profile of the same name attached to `i-0c8898bf51c280eb4` (`AmazonEC2ContainerRegistryReadOnly`), so the host can `aws ecr get-login-password` against IMDS. - Phase 1 latency baseline: ran the full 1/5/10/25/50/100-user sweep against `i-0c8898bf51c280eb4` (instance restarted, container auto-came-up via `--restart unless-stopped`, sweep run from laptop, instance stopped after). Numbers landed in `docs/benchmarks.md`: single-request p99 **860 ms**, throughput saturates at **~7 req/s** with no batching, p99 explodes linearly past saturation (27 s at 100 users). Zero failures across all levels. This is the diagnostic shape Phase 2 targets — ONNX/INT8 to lower the floor, dynamic batching to lift the throughput ceiling. **Phase 1 exit criteria met.**
- Phase 2 (Opt 1) ONNX backend — accuracy parity: refactored `moderation_engine.model` into a backend-agnostic protocol + `build_classifier` factory backed by concrete `PyTorchToxicityClassifier` and `ONNXToxicityClassifier` classes under `moderation_engine/backends/`. `scripts/export_onnx.py` exports `unitary/toxic-bert` to ONNX via `optimum` (438 MB fp32, gitignored). `scripts/parity_check.py` runs both backends on the same N rows and asserts max abs prob diff ≤ tolerance. `colab/eval_onnx.ipynb` ran the full 63,978-row Jigsaw eval on Colab T4 (CUDAExecutionProvider). **Macro-F1 0.6101 — byte-identical to PyTorch at four decimals**; full-set element-wise diff against `baseline_predictions.parquet` shows max 3.76e-06, mean 1.13e-08, 0 rows over the 1e-4 tolerance. ONNX is a lossless numerical equivalent — any latency win is free.
- Phase 2 (Opt 1) ONNX backend — latency sweep: locked Phase 1 sweep (1/5/10/25/50/100 users × 60 s) re-run against the ONNX-backed container on `i-0c8898bf51c280eb4`. Pushed multi-arch image to ECR as `phase2-onnx` + `latest` (amd64 521 MB compressed, down from 873 MB on `phase1-baseline`), pulled on EC2, swapped containers, ran sweep from laptop, stopped instance after. Headline: single-request **p50 −25%** (160 → 120 ms), **throughput +30%** (4.3 → 5.6 req/s at 1 user; +24-40% across the curve, peaking at ~8.8 req/s vs Phase 1's ~7.1). **Surprise**: p99 *widened* at mid concurrency (notably +90% at 25 users) — root cause is ONNX Runtime's CPU provider defaulting to `intra_op_num_threads = num_cores`, which is a win at a single in-flight request but causes thread contention when multiple requests are in flight on a 2-vCPU host. PyTorch was effectively single-threaded per inference (slower p50, tighter tail). The throughput ceiling stays single-process-bound — Opt 3 (dynamic batching) is what lifts it. Zero failures across all 6 levels. Numbers + delta table + "what surprised" written into `docs/benchmarks.md`; optimization-journey row "+ ONNX Runtime" now populated. Phase 1 locust CSVs archived under `docs/locust/phase1-baseline/` for diff. **Phase 2 Opt 1 complete.**
- Phase 2 (Opt 2) INT8 dynamic quantization — accuracy eval: `scripts/quantize_onnx.py` applies `onnxruntime.quantization.quantize_dynamic` (`QInt8`, weights only) to the fp32 ONNX export, producing `models/onnx-toxic-bert-int8/` (105 MB, **−76%** vs 439 MB fp32). `scripts/eval_baseline.py` gained `--onnx-dir` + `--out-prefix` flags plus sort-by-token-length so most batches use minimal padding (3.6× throughput on the 320-row check). Plan called for Colab T4 — didn't work: ORT has no CUDA kernel for `MatMulInteger` / `DynamicQuantizeLinear`, so QInt8 ops fall back to CPU per-op with constant CPU↔GPU tensor copies (first run estimated 13 h). Colab CPU was worse (77 s/batch) — free-tier Xeons predate AVX-VNNI, so INT8 matmul gets no SIMD acceleration. Ran locally on M3 Pro instead (ARM SDOT is the ARM-side AVX-VNNI equivalent): 76.5 samples/s, full 64k eval in 14 min. **Macro-F1 0.6101 → 0.6146 (+0.0045)**; F1 up on 5/6 labels, `severe_toxic` −0.0027 (367 positives, inside noise). Full-set decision diff against `baseline_predictions.parquet`: **1,939 / 383,868 binary decisions flip at thr 0.5 (0.505%)**, 1,858 / 63,978 rows have any flip (2.90%). Mechanism: precision up and recall down on every label — INT8 mildly compresses logits toward 0, FP32 baseline was precision-limited, so the trade-off lands on the right side of the bias/variance curve. Discard branch in plan.txt does not trigger. Deleted `colab/eval_int8.ipynb` (Colab dead end). Saved `onnx_int8_cpu_only` memory so this hardware-feature mismatch doesn't bite again.
- Phase 2 (Opt 2) INT8 dynamic quantization — latency sweep: same locked protocol against the INT8-backed container on `i-0c8898bf51c280eb4`. Multi-arch image pushed to ECR as `phase2-int8` + `latest` (**173 MB compressed, −67% vs `phase2-onnx`**), pulled on EC2, container swapped, sweep run from laptop, instance stopped after. Headline: **p50 −50 to −66% across the entire concurrency curve** (e.g., 1u 120→60 ms, 25u 1,900→640 ms), **p99 −43 to −63%** (1u 930→430 ms, 25u 19,000→7,000 ms), **throughput +120 to +191%** (1u 5.6→12.3 req/s; ceiling lifts from ~9 to ~23 req/s). The ONNX FP32 p99 regression at mid concurrency is gone — at 25 users, ONNX FP32 had p99 +90% vs PyTorch baseline; INT8 turns that into p99 −30% vs baseline at the same level. The threading model didn't change; each request just finishes ~3× faster so contention costs are paid in a much smaller window. Zero failures across all 6 levels. Plan expected "2-3x latency reduction" — actual win is bigger than that on every recorded column. Container size compounds the Opt 1 win: `phase1-baseline` 873 MB → `phase2-onnx` 521 MB → `phase2-int8` 173 MB (cumulative −80%). Opt 1 CSVs archived under `docs/locust/phase2-onnx/`, INT8 CSVs under `docs/locust/phase2-int8/`. Numbers + delta table + "what surprised" written into `docs/benchmarks.md`; optimization-journey row "+ INT8 dynamic quantization" now populated. **Phase 2 Opt 2 complete.**
- Phase 2 (Opt 3) dynamic batching — implemented, measured, bypass wins on 2-vCPU. From-scratch `moderation_engine/batcher.py` with `asyncio.Queue` + a single background worker (no `mosec`/`litserve`); `BATCHING_WINDOW_MS` env var controls window size, `BATCHING_BUCKETS` controls per-bucket length caps, `BATCHING_MAX_BATCH_SIZE` caps batch size. Backends gained a `predict_batch(texts) -> list[dict]` method and `predict(text)` now delegates so there's a single inference codepath. EC2 window sweep at 10u against `i-0c8898bf51c280eb4` ran twice. **Round 1 — naive single-queue batching: 4× throughput regression** vs window=0 single-sample mode (5.4 vs 21.8 req/s, p50 350→1,400 ms). Root cause diagnosed as heavy-tailed Jigsaw seq_len + `padding=True` — locust sample distribution has median 51 / mean 84 / p99 = 512 tokens, so a random batch of 10 hits a long outlier ~40% of the time and all 10 callers pay the 512-token compute. **Round 2 — length-bucketed batching: padding term gone, but bypass still wins.** Per-bucket FIFO queues (defaults `[64, 256, 512]`, sized against the 58%/37%/5% traffic split) drained by a single worker via `asyncio.wait(FIRST_COMPLETED)`, so every batch is homogeneous in length. Recovers most of the regression (14 vs 20.7 req/s at window=2, vs 5.4 naive) but never beats bypass on this hardware. Remaining gap diagnosed as **ORT intra-op saturation**: `CPUExecutionProvider` defaults to `intra_op_num_threads = num_logical_cores` (=2 on `c6i.large`), so a single batch=1 inference already pegs both vCPUs — batching adds queueing without adding compute capacity. The naive 5.4 vs bucketed 14 vs bypass 20.7 curve quantifies "padding waste cost" (8.6 req/s) and "intra-op saturation cost" (6.7 req/s) as separable terms. Production default flipped to `BATCHING_WINDOW_MS=0` in `config.py` so the image ships bypass-by-default; bucketed path stays in-tree (and behind the env var) for the multi-vCPU instance configuration that this $25/mo budget tier doesn't run on. Multi-arch image pushed to ECR as `phase2-batching` + `latest` (173 MB compressed, ~unchanged from `phase2-int8` because only Python source changed). Sweep was cut short at windows 10/20 by what looked like host-level memory pressure under sustained 10u load (container vanished, SSH dropped) — three data points were already enough to call the trend. Sub-percent error rate observed in the bucketed path (4/829 at w=2, 5/768 at w=5) is a small bug not run to root cause this round — `BATCHING_WINDOW_MS=0` sidesteps it for production. Tests cover bucket routing, isolation (short/long never share a batch), per-bucket coalescing, max-batch cap, window=0 bypass, and exception propagation; 11 tests pass under the INT8 backend. New harness: `scripts/run_window_sweep.sh` (single-window locust at fixed concurrency) + `scripts/summarize_window_sweep.py`. Optimization-journey row "+ Dynamic batching (bucketed)" now populated in `docs/benchmarks.md` with the full naive + bucketed tables and the "padding-waste + intra-op-saturation" breakdown. **Phase 2 Opt 3 complete — code merged, default behaviour is bypass.**
- Phase 2 (Opt 4) ORT threading tune — direct test of the Opt 3 intra-op-saturation hypothesis. `backends/onnx.py` gained an `intra_op_num_threads` constructor param exposed via the `ONNX_INTRA_OP_THREADS` env var (default flipped to `1` after the sweep — see below). `batcher.py` restructured from "one worker draining all buckets via `asyncio.wait(FIRST_COMPLETED)`" to **one `asyncio.Task` per bucket**, so two buckets can submit two inferences to the default executor in true parallel. New test `test_buckets_run_concurrently` asserts the parallelism contract directly (two sleep=0.15s batches in different buckets finish in <0.25s wall). 4-config sweep against `i-0c8898bf51c280eb4` at 10u × 60 s: `intra_op=2/w=0` 22.0 req/s (control, reproduces Opt 2 baseline), **`intra_op=1/w=0` 23.6 req/s (+7.4%, p99 2,100→2,000 ms)**, `intra_op=1/w=2` 17.7 req/s (−19%), `intra_op=1/w=5` 17.1 req/s (−22%). Headline: **the threading tune wins on its own, the batcher still loses even with intra-op contention removed.** Root cause of the batcher loss is bucket-traffic imbalance — Jigsaw distribution puts 58% of traffic in the 64-token bucket and 5% in the 512-token bucket, so the second bucket worker idles most of the time and the parallelism contract is unused; meanwhile single-bucket batches give up the intra-op parallelism that bypass enjoys (`intra_op=1` × batch=N ≈ N × single-inference time). The two changes have to be evaluated together — they don't stack. Same sub-percent error rate (5/1049 and 2/1012) in the bucketed path as the Opt 3 round-2 sweep; bypass default sidesteps it. **Production default `ONNX_INTRA_OP_THREADS=1`** in `config.py`, layered on top of `BATCHING_WINDOW_MS=0`. Cumulative win over the Phase 1 baseline at 10u: p99 4,800→2,000 ms (**−58%**), throughput 7.0→23.6 req/s (**+237%**); on 1u the floor is 860→~400 ms (−54%). Sweep CSVs saved under `docs/locust/threading/`. Multi-arch image `phase2-threading` + `latest` in ECR (~173 MB, ~unchanged from `phase2-batching` since only Python source changed). New harness: `scripts/run_threading_sweep.sh`. Optimization-journey row "+ ORT threading tune" populated in `docs/benchmarks.md` with the 4-config table and the two-cause analysis of the batcher loss. **Phase 2 Opt 4 complete — code merged, default behaviour is intra_op=1 bypass.**
