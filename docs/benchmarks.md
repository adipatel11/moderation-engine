# Benchmarks

This file is the single source of truth for the project's measured numbers. Every optimization in Phase 2 is appended here against a locked protocol established in Phase 1.

## Protocol (locked in Phase 1 — do not change once set)

- **Hardware**: AWS EC2 `c6i.large` (2 vCPU compute-optimized, x86_64), `us-east-1`
- **Test inputs**: Jigsaw Toxic Comment Classification test set (realistic mix of toxic and non-toxic, never a single repeated string)
- **Load generator**: `locustfile.py`, concurrency sweep at 1, 5, 10, 25, 50, 100 users
- **Metrics recorded**: p50 / p95 / p99 latency (ms), throughput (req/s), error rate (%)
- **Accuracy metrics**: per-label F1, precision, recall, ROC-AUC on Jigsaw test set

## Accuracy floor (Phase 1)

Generated by `scripts/eval_baseline.py` against the 63,978-row Jigsaw scored test split at threshold 0.5. Raw per-row predicted probabilities are committed to [`baseline_predictions.parquet`](baseline_predictions.parquet) so future optimization runs (ONNX, INT8) can be diff'd against this snapshot without re-running the PyTorch baseline. Full per-label breakdown lives in [`baseline_accuracy.md`](baseline_accuracy.md).

Macro-average F1: **0.6101** (recall-heavy at threshold 0.5 — the model ranks well, ROC-AUC ≥ 0.97 on every label).

### ONNX accuracy parity (Phase 2, Opt 1)

Eval re-run against the ONNX-exported model on Colab T4 via `colab/eval_onnx.ipynb`. Per-label F1 / precision / recall / ROC-AUC are byte-identical to the PyTorch baseline at four decimals; full per-label breakdown in [`onnx_accuracy.md`](onnx_accuracy.md). Raw probabilities in [`onnx_predictions.parquet`](onnx_predictions.parquet).

Element-wise parity against `baseline_predictions.parquet` on all 63,978 rows × 6 labels (~384k probability comparisons):

- overall max abs diff: **3.76e-06**
- overall mean abs diff: **1.13e-08**
- rows exceeding the 1e-4 tolerance: **0** (every label)

Macro-F1 unchanged at **0.6101**. Conclusion: ONNX export is a lossless numerical equivalent of the PyTorch baseline — any latency win is a free win.

## Latency / throughput

Closed-loop concurrency sweep generated by `scripts/run_locust_sweep.sh`
(60 s per level, zero think time, 1000-row seed-42 Jigsaw sample). Full
per-level locust CSVs live under `docs/locust/` (gitignored — re-run the
sweep to regenerate). Markdown rendering of this same table lives at
[`locust_summary.md`](locust_summary.md).

| Users | Requests | Failures | p50 (ms) | p95 (ms) | p99 (ms) | Throughput (req/s) | Error rate | Notes |

|     1 |      252 |        0 |      160 |      600 |      860 |                4.3 |      0.00% | Single-request floor |
|     5 |      421 |        0 |      510 |    1,700 |    3,100 |                7.1 |      0.00% | Throughput peak; no batching |

|    25 |      372 |        0 |    3,300 |    7,400 |   10,000 |                6.3 |      0.00% | Queue-bound |
|    50 |      368 |        0 |    6,800 |   13,000 |   18,000 |                6.2 |      0.00% | Queue-bound |
|   100 |      334 |        0 |   15,000 |   22,000 |   27,000 |                5.7 |      0.00% | Queue-bound |

The throughput ceiling at ~7 req/s and the linear p99 explosion past
saturation are the diagnostic shape Phase 2 attacks: ONNX Runtime should
move the single-request floor down, INT8 quantization should move it
down again, and the dynamic batcher is what lets throughput actually
grow with concurrency instead of saturating against single-threaded
PyTorch inference.
